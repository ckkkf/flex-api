   2.1 分支分类
   我们将分支分为三类：长期分支、短期分支 和 发布/修复分支。
   2.1.1 长期分支
- main (或 master)：生产环境分支。仅存放经过严格测试、可直接发布的代码。禁止直接在 main 上提交代码。该分支只能通过合并，禁止推送。
- develop：开发主分支。所有开发功能的分支最终都要合并到这里，它是项目的“集成中心”。禁止在dev分支直接开发，
  2.1.2 短期分支 (功能开发)
- feature/*：新功能分支。从 develop 检出，命名如 feature/login-logic。开发完成后合并回 develop 并删除。
  2.1.3 维护分支 (修复与发布)
- hotfix/*：紧急修复分支。直接从 main 检出，用于修复生产环境的严重 Bug。修复后需同时合并回 main 和 develop。
- release/*：发布预览分支。版本发布前从 develop 检出进行最后的 QA 测试和版本号修改。

2.2 开发流程
开发步骤如下：
1. 拉取 远程develop分支 代码，基于 本地develop分支 创建 本地feature/*
2. 在 本地feature/*分支 上开发，开发完毕后提交，先不要推送
3. 先切换到 本地develop分支 拉取远程develop分支 的代码
4. 然后再切回 本地feature/*分支，将 本地develop分支 合并到 本地feature/*分支
5. 处理冲突
6. 处理冲突完毕后，推送 本地feature/*分支 到远程
7. 去Github提交PR合并 远程feature/*分支 到 远程develop分支

3. 开发规范--代码编写规范
   3.1 数据传递
   数据需要通过以下对象传递：
   前端 <--> Request <---> VO <---> DTO <---> PO <---> 数据库
1. 总体规范：
1. 实体类请使用lombok的@Data
2. 每个字段使用如下注释：
   /**
* xxx  
  */
3.
2. 详细规范：
1. PO：必须和数据表1:1映射
2. DTO：用于服务间信息传递
3. VO：敏感字段请删除或脱敏
4. Request：请使用JSR303进行校验，额外规则如下：
   1. String类型可使用@NotBlank（此注解禁止为null、空、空格）
   2. 其余类型自行选择

3.2 数据库查询
1. 尽量使用分页查询代替数据列表查询，数据量过多禁止一次全部查出来，必须使用分页查询。
2. 禁止在循环中操作数据库，请替换为批量操作数据库的api。
3. 删除必须使用逻辑删除，通过逻辑删除标记 deleted_at 进行标记：
1. 未删除：空
2. 已删除：在 deleted_at 存入删除时间，例如：2026-04-16 14:36:52.504493 +00:00
   通过调用mybatis-plus的删除，如果配置后，其会自动逻辑删除。

3.3 Controller
1. 必须通过统一的对象包装后返回给前端。
2. 禁止在controller中书写业务代码。
3. 必须controller层打印接口调用日志，格式为：
   log.debug("【xxx服务接口】接口描述信息，参数1: {}, ... , 参数n: {}", 参数1, ... , 参数n)
   例如：
   log.debug("【用户地址服务】查询地址信息。userId：{}", StpUtil.getLoginId());

3.4 Service
1. 可通过JSR303进行参数校验，使用后务必测试校验是否生效。

3.5 yml配置
项目yml配置分为以下类别：
- application.yaml 配置模板
- application-dev.yaml 开发环境配置
- application-test.yaml 测试环境配置
- application-prod.yaml 生产环境配置
